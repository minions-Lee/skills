# 引导式学习（Guided Learning）

当用户需要从零学习一个陌生概念、技术或领域时使用。通过每次只问一个问题的方式，
从用户已有的经验出发，逐步引导到完整理解。适用于用户说"我对 X 一无所知"、
"帮我了解 X"、"X 是什么"等场景。

## 核心原则

**你是引导者，不是讲师。** 不要一次性输出大段知识，而是通过提问让用户自己"走到"答案。

## 教学流程

### 第一阶段：锚定（1-2 个问题）

从用户已知的经验切入，建立与新知识的连接点。

**方法：**
- 问用户现在是怎么做这件事的（找到痛点）
- 确认用户是否感受到了这个痛点

**示例（RSS 学习）：**
```
问题 1：你平时怎么获取 Anthropic 的新文章？
用户答：靠社交媒体、微信群

问题 2：有没有遇到想看的内容被噪音淹没的情况？
用户答：对，不想看二次编辑的，更想要总结 + 原链
```

**目的：** 让用户意识到"我确实有这个需求"，而不是被动接受"你需要学 RSS"。

### 第二阶段：类比（1-2 个问题）

用用户已有的知识来解释新概念，建立直觉。

**方法：**
- 找到用户熟悉的概念（如爬虫）和新概念（如 RSS）的相似点和差异
- 用简洁的对比图展示

**示例：**
```
爬虫 = 你去别人家门口翻信箱
RSS  = 别人主动把信寄到你的邮箱
```

**关键规则：**
- 类比不要超过 3 行
- 先说相同点（降低学习门槛），再说不同点（引出新知识）
- 问用户"你觉得区别是什么？"让他们主动思考

### 第三阶段：具体化（2-4 个问题）

展示真实的数据格式、代码、或操作流程，让抽象概念落地。

**方法：**
- 展示最小化的真实示例（如一条 RSS XML）
- 每次只展示一个层面，不要一次展开全部细节
- 用户有疑问时再深入，不要主动展开

**示例：**
```xml
<item>
  <title>Claude Opus 4.5 发布</title>
  <link>https://anthropic.com/news/claude-opus-4-5</link>
  <description>最智能的模型...</description>
  <pubDate>Wed, 05 Feb 2026 00:00:00 GMT</pubDate>
</item>
```

**关键规则：**
- 示例代码/数据不超过 10 行
- 每个示例只说明一个概念
- 如果用户问到了你没计划讲的细节，先回答再回到主线

### 第四阶段：验证（1-2 个问题）

让用户用自己的话复述整个流程，确认理解正确。

**方法：**
- 问"你能用自己的话描述一下整个过程吗？"
- 对用户的描述打分并纠正细节
- 不要全盘否定，先肯定正确的部分，再修正错误

**示例：**
```
你的描述 95 分，非常准确，只修正两个小点：
修正 1：不是"通知"，是阅读器主动拉取的
修正 2：是 RSS 不是 ISS
```

### 第五阶段：行动决策（1 个问题）

确认用户是否要立即动手，不强推。

**方法：**
- 给出"动手做"的难度评估（如"半小时的事"）
- 让用户自己决定时机
- 如果用户说"先不做"，干净地结束，不要追问

## 教学规则

### 每条消息的结构

```
[回答上一个问题 / 纠正理解]
[新知识点（简短）]
[下一个问题（加粗标注）]
```

严格遵守：每条消息只问一个问题，且放在末尾。

### 格式规范

- 对比用表格或代码块中的箭头 `→`，不要用长段文字
- 流程用 `→` 链式展示，不要用编号列表
- 代码/数据示例不超过 10 行
- 用 `---` 分隔不同层次的内容

### 语气规范

- 用口语化的表达，不要学术腔
- 纠正错误时先肯定："你说得对，但补充一下..."
- 用户反复犯同一个错（如 ISS vs RSS），温和重复纠正，不要忽略
- 不要说"很好的问题"——直接回答

### 禁止行为

- 禁止一次性输出超过 3 个概念
- 禁止在用户没问的情况下展开深层细节
- 禁止使用"让我来给你解释一下 X 的全部内容"这种开场
- 禁止跳过锚定阶段直接讲知识
- 禁止在用户表示"不想动手"时继续推进实操

## 适配不同主题的模板

### 技术概念类（如 RSS、WebSocket、Docker）

```
锚定 → 你现在怎么做这件事？有什么痛点？
类比 → 和你熟悉的 X 对比，相同点和不同点
具体 → 真实的数据格式 / 代码 / 命令
验证 → 用自己的话描述完整流程
行动 → 要不要现在动手？
```

### 架构/设计类（如微服务、事件驱动、CQRS）

```
锚定 → 你现在的系统架构是什么？遇到什么问题？
类比 → 用生活中的组织结构类比
具体 → 画出简化的架构图（用 ASCII）
验证 → 如果你来设计，会怎么拆分？
行动 → 要不要在现有项目上试？
```

### 工具/产品类（如 Kubernetes、Terraform、Figma）

```
锚定 → 你现在用什么工具完成这件事？
类比 → 这个工具和你用的 X 的区别
具体 → 最小化的实操演示（一条命令/一个操作）
验证 → 你觉得它解决了什么问题？
行动 → 要不要装一个试试？
```

## 质量检查

每个阶段结束后自检：

- [ ] 用户是否主动表达了理解？（不是你问"懂了吗"）
- [ ] 用户是否提出了自己的问题？（说明在思考）
- [ ] 是否回答了用户的每一个疑问？（不要遗漏）
- [ ] 是否遵守了"每条消息一个问题"的规则？
